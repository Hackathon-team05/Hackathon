# Unity - はじめに

## ようこそ

おめでとうございます。
これで、あなた自身のビットの山を手に入れました。
この1と0の集まりを使って何をしますか？
この文書は、その答えを決める手助けとなるはずです。

Unityはユニットテストフレームワークです。
小さく、実用的に保つことを目標としています。
Unityの中核となるテストフレームワークは、1つのCファイルと2つのヘッダーファイル、合計3ファイルで構成されています。
これらが連携し、テストを容易にする関数とマクロを提供します。

Unityはクロスプラットフォームで動作するよう設計されています。
C標準に準拠しつつ、規則に独自の解釈を加える多数の組み込みCコンパイラーもサポートできるよう配慮されています。
UnityはGCC、IAR、Clang、Green Hills、Microchip、MS Visual Studioなど、多くのコンパイラーで利用されています。
新しいターゲットに対応させるための作業も、それほど多くありません。

### 文書の概要

#### Unityアサーションリファレンス

Unityが提供するすべてのアサーションについて説明します。
アサーションはユニットテストの中心となる機能です。
Unityのほかのどの機能よりも、アサーションを使う時間が長くなるでしょう。

#### Unityアサーション早見表

前述の文書で説明されているアサーションを短くまとめたものです。
印刷して手元に置き、Unityの機能に慣れるまで参照するのに適しています。

#### Unity設定ガイド

新しいターゲットやコンパイラーでUnityを使用するときに参照する文書です。
設定項目を説明し、要件に合ったテスト環境へカスタマイズする手助けをします。

#### Unityヘルパースクリプト

テスト作業を簡略化するために利用できるヘルパースクリプトについて説明します。
Unityのインストール先にある`auto`ディレクトリに含まれる、任意利用のRubyスクリプト群を扱います。
Unityを使うためにRubyやこれらのスクリプトが必要なわけではありません。
必要な人のための便利な機能として提供されています。

#### Unityライセンス

ライセンスファイルのないオープンソースプロジェクトはありません。
この短い文書では、本ソフトウェアを使用する際に同意する条件を説明します。
基本的には、どのような用途でも役立ててほしいと考えていますが、問題が起きても開発者を責めないでください、という内容です。

### フォルダーの概要

GitHubなどからUnityを入手した場合、目の前に突然現れた大量のファイルに驚くかもしれません。
心配はいりません。Unity自体は非常に小さいものです。
残りのファイルは、作業を楽にするために用意されています。
無視しても、必要に応じて利用しても構いません。
プロジェクト内の各フォルダーの概要は次のとおりです。

- `src` - 最も重要なコードです。このフォルダーには1つのCファイルと2つのヘッダーファイルがあります。
  この3ファイルがUnityそのものです。
- `docs` - 今この文書を読んでいるなら、すでにこのフォルダーを見つけているかもしれません。
  便利な文書はすべてここにあります。
- `examples` - Unityの使用例がいくつか含まれています。
- `extras` - Unityの中核には含まれない、任意利用の追加機能です。
  James Grenningの書籍を通じてUnityを知った場合は、ここを確認してください。
- `test` - Unity本体とスクリプトをテストするためのものです。
  Unityを利用するだけなら、このフォルダーを開く必要はほとんどありません。
  Unityを新しいツールチェーンへ移植する担当者になった場合は、すべてが正しく設定されていることを確認するのに適した場所です。
- `auto` - テスト作業を簡略化する便利なRubyスクリプトがあります。
  これらは完全に任意であり、Unityの利用に必須ではありません。

## テストファイルの作成方法

テストファイルはCファイルです。
通常は、テスト対象となるCモジュールごとに1つのテストファイルを作成します。
テストファイルでは`unity.h`と、テスト対象Cモジュールのヘッダーをインクルードします。

次に、テストファイルへ`setUp()`関数と`tearDown()`関数を追加します。
`setUp()`には各テストの前に実行したい処理を記述できます。
`tearDown()`には各テストの後に実行したい処理を記述できます。
どちらの関数も引数を取らず、値を返しません。
必要がなければ、一方または両方を空のままにできます。

Ceedlingまたはテストランナー生成スクリプトを使用している場合は、これらを完全に省略できることがあります。
判断できない場合は、まず試してください。
リンク時にコンパイラーが`setUp`または`tearDown`を見つけられないと報告したら、少なくとも空の関数を追加する必要があります。

ファイルの大部分は、一連のテスト関数になります。
テスト関数は、名前を`test_`または`spec_`で始める慣例があります。
必須ではありませんが、ほかの開発者がテスト関数だと判断しやすくなります。
また、UnityやCeedling付属の自動化スクリプトも、既定ではこの接頭辞を持つテスト関数を検索します。
テスト関数は引数を取らず、値を返しません。テスト結果の集計はUnity内部で処理されます。

最後に、テストファイルの末尾へ`main()`関数を記述します。
この関数では`UNITY_BEGIN()`を呼び出し、各テストに対して`RUN_TEST`を呼び出し、最後に`UNITY_END()`を呼び出します。
これによって各テスト関数が実際に実行されるため、すべての関数に個別の`RUN_TEST`呼び出しが必要です。

各テストを`main()`関数へ追加し続けるのは面倒になることがあります。
ビルド処理でヘルパースクリプトを利用できるなら、便利な[generate_test_runner.rb][]スクリプトの使用を検討してください。
推奨される命名規則に従っていれば、このスクリプトが`main()`関数とすべての呼び出しを生成します。
この場合、テストファイルに`main()`関数を記述する必要はありません。

完成したテストファイルは、次のようになります。

```C
#include "unity.h"
#include "file_to_test.h"

void setUp(void) {
    // ここで事前準備を行う
}

void tearDown(void) {
    // ここで後処理を行う
}

void test_function_should_doBlahAndBlah(void) {
    // テスト処理
}

void test_function_should_doAlsoDoBlah(void) {
    // 追加のテスト処理
}

// generate_test_runner.rbを使用する場合は不要
int main(void) {
    UNITY_BEGIN();
    RUN_TEST(test_function_should_doBlahAndBlah);
    RUN_TEST(test_function_should_doAlsoDoBlah);
    return UNITY_END();
}
```

いずれ、これ以上のカスタマイズが必要になる場合もあります。
その場合は設定ガイドを参照してください。
ただし、ここまでの内容でテストを始めるには十分です。

### テスト関数の実行

テストランナー用の`main()`関数を自分で記述する場合、テストを実行する方法は2つあります。

従来の形式は次のとおりです。

```c
RUN_TEST(func, linenum)
```

または、関数の先頭から開始する簡略版を使用できます。

```c
RUN_TEST(func)
```

これらのマクロは、テスト呼び出し前に必要な準備を行い、呼び出し後に後処理と結果集計を行います。

### テスト関数の無視

テストが未完成、または何らかの理由で無効な場合があります。
そのような場合は`TEST_IGNORE`を呼び出せます。
制御は直ちにテストの呼び出し元へ戻り、失敗は報告されません。
テストランナーを自動生成している場合に便利です。

```c
TEST_IGNORE()
```

このテストを無視し、直ちに戻ります。

```c
TEST_IGNORE_MESSAGE(message)
```

このテストを無視し、直ちに戻ります。
テストを無視した理由を示すメッセージを出力します。

### テストの中断

エラー条件によってテストが無限ループに入る場合や、残りの処理を実行せずにテストを早期終了したい場合があります。
Unityでは、2つのマクロがこの機能を提供します。
最初の`TEST_PROTECT`は機能を準備し、緊急中断を処理します。
その後、テスト内の任意の場所で`TEST_ABORT`を使用すると、最後に呼び出した`TEST_PROTECT`へ戻れます。

```c
TEST_PROTECT()
```

準備および捕捉用のマクロです。

```c
TEST_ABORT()
```

テスト中断用のマクロです。

例:

```c
main()
{
    if (TEST_PROTECT())
    {
        MyTest();
    }
}
```

`MyTest`が`TEST_ABORT`を呼び出すと、プログラムの制御は直ちに`TEST_PROTECT`へ戻り、戻り値は0になります。

## テストファイルのビルドと実行方法

少なくともCやC++のような言語では、新しいユニットテストフレームワークを使い始める際の最大の難関がここです。
これらの言語はハードウェアに近い処理を非常に得意とします（「金属に近い」という表現より、「シリコンに近い」のほうが正確かもしれません）。
通常は長所となるこの特性も、テストを難しくする場合があります。

ツールチェーンには2つの有力な選択肢があります。
これまでの経験によっては、どちらも実機上でユニットテストを実行しないことに驚くかもしれません。
理由は多数ありますが、要点は次のとおりです。

- 実機上では、処理能力やメモリーなどの制約が多すぎる。
- 実機上では、すべてのレジスターを完全には制御できない。
- 実機上でのユニットテストは、より難しい。
- ユニットテストとシステムテストは異なるため、分離すべきである。

実機でテストを実行する代わりに、多くの開発者はネイティブアプリケーション（gccやMSVCなどを使用）またはシミュレーター上で動くアプリケーションとしてテストを開発します。
どちらも有効な選択肢です。
ネイティブアプリケーションには、高速でセットアップしやすいという利点があります。
シミュレーター上のアプリケーションには、ターゲットアプリケーションと同じコンパイラーを使用できるという利点があります。
これらの設定方法は設定ガイドで説明されています。

どちらを動作させる場合でも、レジスターセットを含むファイルにいくつか変更が必要になる可能性があります（後述）。

いずれの場合も、Unity、テストファイル、およびテスト対象のCファイルをリンクしてテストをビルドします。
これらのファイルから、そのモジュールのテスト一式として実行できる実行ファイルが作成されます。
次のテストファイルについても、この処理を繰り返します。
テストを個別の実行ファイルに分離できるため、システムをより徹底してユニットテストでき、最終リリースへテストコードが混入することも防げます。
