# SKILL: システム設計書作成支援

## 概要
基本設計（外部設計）・詳細設計（内部設計）の設計書を
Claudeが作成・添削・改善提案するためのスキル。

## トリガー条件
以下のような依頼でこのスキルを使う：
- 「設計書を書いて／直して／レビューして」
- 「基本設計／詳細設計のドキュメントを作りたい」
- 「設計書のどこを改善すればよいか教えて」
- 「システム構成図やER図を整理したい」

---

## 設計フェーズの定義

| 比較項目 | 基本設計（外部設計） | 詳細設計（内部設計） |
|----------|----------------------|----------------------|
| 設計の視点 | ユーザー視点：システムが何を提供するかを定義 | 開発者視点：どのロジックで実現するかを定義 |
| 目的 | **「何を実現するか（What）」**を決定 | **「どのように実現するか（How）」**を決定 |
| 抽象度 | 高い（概念的・俯瞰的） | 低い（具体的・実装可能レベル） |
| 主な対象者 | クライアント、エンドユーザー、業務担当者 | プログラマー、実装チーム |
| 主な成果物 | システム構成図、機能一覧、画面遷移図、ER図（論理） | クラス図、シーケンス図、処理フロー、DB物理設計書 |

---

## ルーブリック（全5項目・各4点）

### 1. システム化の目的と要件

| ランク | 基準 |
|--------|------|
| A (4点) | ビジネス課題・ユースケース・機能分解（FBS）が根拠とともに網羅されている |
| B (2〜3点) | いずれかは記載されているが根拠・粒度が不十分 |
| C (0〜1点) | 目的・要件・ユースケースのいずれも記載なし |

**作成支援のポイント：**
- 解決するビジネス課題・業務フローを具体的に述べる
- ユーザーがシステムをどう利用するか（ユースケース図）を定義する
- 要求は「理由」とセットで整理し、仕様との混同を避ける（USDM形式推奨）
- 機能をユーザー視点で要素分解した機能分解構造（FBS）リストを作成する

---

### 2. 基本設計（外部設計）の記述

| ランク | 基準 |
|--------|------|
| A (4点) | 画面設計・データモデル（論理）・外部I/F仕様が図つきで十分に記述されている |
| B (2〜3点) | 図はあるが説明が不明瞭、または一部の要素が欠けている |
| C (0〜1点) | 画面設計・ER図・外部I/Fのいずれも記載なし |

**作成支援のポイント：**
- **画面設計**：画面一覧・遷移図・ワイヤーフレームを作成し、入力制限（必須・型・文字数）とエラー表示条件を明記する
- **データモデル（論理設計）**：業務観点でエンティティと関係性を示すER図を含める
- **外部I/F仕様**：他システムとの連携タイミング・データフォーマット・エラー時のリカバリ方法を定義する

---

### 3. 詳細設計（内部設計）の記述

| ランク | 基準 |
|--------|------|
| A (4点) | PBS・UMLモデル・クラス図・詳細ロジックが実装可能なレベルで記述されている |
| B (2〜3点) | 図はあるが異常系・物理設計の詳細が不十分で実装に迷う箇所がある |
| C (0〜1点) | 内部ロジック・DB物理設計・クラス構造のいずれも記載なし |

**作成支援のポイント：**
- **PBS（製品分解構造）**：機能を「画面・API・DBテーブル等」のサブ成果物に対応付ける
- **動的挙動（UMLモデル）**：シーケンス図（オブジェクト間のメッセージ）・状態遷移図を作成する
- **プログラム構造（クラス図）**：属性・操作・クラス間の関係（継承・集約・依存）を定義する
- **詳細ロジック**：正常系だけでなく異常系の処理フロー、DBのインデックス・データ型などの物理設計を含める

---

### 4. 図・表・ダイアグラム

| ランク | 基準 |
|--------|------|
| A (4点) | 必須図表が揃い、文中で言及されており、読者がゼロから再現できる粒度 |
| B (2〜3点) | 図表は揃っているが一部不明瞭・説明が不十分 |
| C (0〜1点) | 必要な図表のいずれかが欠けており、説明も不十分 |

**作成支援のポイント：**
- 必須図表チェックリスト：
  - [ ] システム構成図（アーキテクチャ図）
  - [ ] ER図（論理設計）
  - [ ] 画面遷移図
  - [ ] シーケンス図 / 状態遷移図
  - [ ] クラス図
  - [ ] DB物理設計書（テーブル定義）
- 図表には通し番号とキャプションをつける
- 本文中で「図1に示すように〜」と必ず言及する
- 図表はMermaid記法またはMarkdownの表で代替し、ユーザーに清書を促す

---

### 5. 標準化ルールと非機能要件

| ランク | 基準 |
|--------|------|
| A (4点) | 命名規約・テンプレート・非機能要件（性能・セキュリティ・可用性）が数値目標つきで記載 |
| B (2〜3点) | 一部は記載されているが数値目標や根拠が不十分 |
| C (0〜1点) | 標準化ルール・非機能要件のいずれも記載なし |

**作成支援のポイント：**
- **標準化方針**：設計書テンプレート・命名規約（変数・関数・DBオブジェクト名）・共通部品の利用ルールを定義する
- **実行環境**：対応OS・ブラウザ・ランタイムバージョンなどの制約を明記する
- **非機能要件**：レスポンスタイム・スループット・セキュリティポリシー・可用性（障害検知手法）を具体的な数値目標で記載する
  - 例：「APIレスポンス1秒以内」「稼働率99.9%以上」「認証はJWT＋HTTPS必須」

---

## Claudeの行動指針

1. **新規作成**：ユーザーからシステムの概要・目的・対象ユーザーをヒアリングし、上記5項目を満たす設計書の草案を生成する。不足情報は質問で補う。
2. **添削・レビュー**：提出された設計書を5項目ごとにランク判定し、改善点を具体的に提示する。
3. **部分改善**：特定セクションのみ改善を求められた場合は、そのセクションのルーブリックに集中して対応する。
4. **トレーサビリティ確保**：要件ID・機能ID・プログラムIDを紐付け、変更影響範囲を追跡できる構造を推奨する。
5. 図表はMermaid記法またはMarkdownの表で代替し、ユーザーに清書を促す。
