# Arduinoシンセサイザー 仕様書

## 概要

サーバーArduinoのスイッチを入れると起動し、かえるの歌が演奏される。  
各楽器Arduinoは楽譜データをフラッシュメモリ（PROGMEM）に保持し、サーバーArduinoから受信するBPMクロックに同期して演奏する。楽器ArduinoはBPMクロックに合わせて音データをサーバーArduinoへSPI送信し、サーバーArduinoがPC（Processing）へUSBシリアル通信で転送する。  
Processingは受信データを解析し、各楽器ごとの音を正弦波の加算合成によって作成、エンベロープをかけたのち、ビブラートやリバーブといったエフェクトを付加して鳴らす。

## ハードウェア構成

### Arduino構成

| 種別 | 台数 | 役割 |
|------|------|------|
| サーバーArduino | 1台 | BPM管理・PC通信・マイク入力・SPI制御 |
| 楽器Arduino（メロディ） | 3台 | 楽曲データ保持・音データ送信 |
| 楽器Arduino（リズム） | 1台 | 楽曲データ保持・音データ送信 |

### PC接続

- サーバーArduinoのみがPCとUSBシリアル通信で接続
- ProcessingがPCで動作し、音声を生成・出力する

### マイク

- サーバーArduino側に設置
- 種類：MAX4466アナログマイクモジュール
- 用途：観客の拍手のテンポ検出

### 圧力センサ

- 楽器Arduino側に設置
- 圧力センサの値が閾値を超えると、その楽器の手動演奏ON（frog_state=1）として扱う
- 圧力センサの値が閾値を下回ると、手動演奏OFF（frog_state=0）として扱う
- 閾値の具体値：**TODO（実機で調整要）**
- カエル人形などの物理オブジェクトを用いた入力は、余裕がある場合の追加要素とする

### 風車入力

- 拍手だけでは独自性が弱いため、風車を用いた入力を追加検討する
- 風車のプロペラを赤外線距離センサで検知し、回転状態を入力として扱う
- 赤外線距離センサでの検知が難しい場合は、プロペラに発電機を接続し、発生電圧を `analogRead` で検知する方式に切り替える
- 風車入力を用いる場合、システム側から演奏開始タイミングを知らせる合図音を鳴らす

## 通信仕様

### サーバー ↔ 楽器Arduino（SPI通信）

- 一対多（1台のサーバーが4台の楽器Arduinoと通信）
- 4線式独立スレーブセレクト方式（SCK/MOSI/MISO + 楽器ごとの個別CSピン）
- CSピン割り当て：楽器1=D4 / 楽器2=D5 / 楽器3=D6 / 楽器4=D7
- SPI設定：1MHz（SPI_MODE0）、テスト後に最大4MHzまで増速

#### SPIフレーム（4バイト、全二重）

1拍ごとにサーバーが各楽器のCSをアサートして4バイトのSPIトランザクションを行う。

| 方向 | バイト0 | バイト1 | バイト2 | バイト3 |
|------|---------|---------|---------|---------|
| サーバー→楽器（MOSI） | `0xFE`（BPMクロック） | `0x00` | `0x00` | `0x00` |
| 楽器→サーバー（MISO） | `pitch` | `velocity` | `frog_state` | `checksum` |

- `pitch`：MIDIノート番号（0=休符）
- `velocity`：音の強さ（0〜127）
- `frog_state`：`0x00`=手動演奏なし / `0x01`=手動演奏中（カエル人形設置）
- `checksum`：`pitch XOR velocity XOR frog_state`
- `instrument_id` はサーバーがどのCSをアサートしたかで識別するため送信不要

#### エラーハンドリング

- チェックサム不一致を1楽器につき**3回連続**検出した場合、フェイルセーフ発動
- フェイルセーフ発動後はその楽器のCSをアサートしない（無音で演奏継続）

### サーバー ↔ PC（USBシリアル通信）

- 通信速度：115200 bps
- サーバーがSPIで受け取った音データを加工してProcessingへ転送する

#### シリアルフレーム（4バイト固定長）

| バイト | フィールド | 内容 |
|--------|-----------|------|
| 0 | `type` | イベント種別 |
| 1 | `instrument_id` | 楽器番号（0〜3） |
| 2 | `pitch` | MIDIノート番号（0=休符） |
| 3 | `velocity` | 音の強さ（0〜127） |

`type` の値：

| 値 | 意味 |
|----|------|
| `0x01` | NoteOn（pitch>0） |
| `0x02` | NoteOff（instrument_idのみ有効、pitch/velocity=0） |
| `0xFF` | Failsafe（instrument_idのみ有効） |

#### サーバーの送信ロジック

```
SPIトランザクション完了後:
  checksum不一致3回連続 → type=0xFF をProcessingへ送信
  frog_state == 0x01    → type=0x02（NoteOff）をProcessingへ送信
  pitch > 0             → type=0x01（NoteOn）をProcessingへ送信
  pitch == 0            → type=0x02（NoteOff）をProcessingへ送信（休符）
```

## 楽曲データ管理

### 格納場所

- **各楽器Arduinoのフラッシュメモリ（PROGMEM）に持たせる**
- 楽器ArduinoはBPMクロックを受信するたびに次の音データを読み出してSPIバッファに積む
- メロディ用（MELODY_DATA）とリズム用（RHYTHM_DATA）を独立した配列で定義し、`INSTRUMENT_ID` のコンパイル時定数で切り替える

## BPM制御

### 基本BPM

- かえるの歌のデフォルトBPMをサーバーArduinoが管理
- BPM変更時はSPIクロック（0xFE送信）の送信間隔を変えることで楽器側の演奏テンポが変化する

### 拍手テンポ同期

- マイクで観客の拍手を検出し、BPMをリアルタイムで変更する
- **ピーク検出**でテンポを検出
- ノイズ除去：移動平均 + 移動標準偏差をリングバッファで計算し、±2σを超えた値をノイズとして排除する
- BPM変化をn拍目で検知した場合、n+1拍目から新しいBPMを適用する
- 毎拍の小さな揺れは無視し、一定以上のBPM変化があった場合のみ反映する
- 拍手入力はBPM変更に限定し、演奏構造の変更は今回の必須仕様には含めない

### 拍手検出の評価目標

- 拍手テンポ検出の誤差許容：暫定5%
- 通信開始から実際に音が再生されるまでの許容レイテンシ：200ms以内
- 反応速度：BPM変化を検知した次拍から反映する

> **未確定事項：** BPM変化とみなす閾値、5%誤差の測定方法、レイテンシ測定方法は要定義

## 起動シーケンス

### 起動フロー

1. サーバーArduinoの電源ON（スイッチ投入）
2. サーバーArduino自己初期化
   - デフォルトBPMを設定
   - SPI・シリアル通信の初期化
3. 各楽器ArduinoへCS信号を順次送信し、接続確認
   - 通常のSPIフレーム（0xFE送信）を200ms間隔で最大3回試行
   - チェックサム一致の返答あり → 「アクティブ」として登録
   - 3回失敗 → スキップして演奏開始（無音パート扱い）
4. PC側Processingが起動済みであることを確認（または確認なしで送信開始）
   - **TODO**：Processingとのハンドシェイクの有無を要決定
5. BPMクロック（0xFE）をアクティブな全楽器Arduinoへ順次送信開始
6. かえるの歌の自動演奏を開始

## Processing（音声生成）

### 音声生成方式

- 正弦波の**加算合成**（各楽器の倍音構成を重ね合わせる）
- エンベロープ（Attack / Decay / Sustain / Release）を適用
- エフェクト：ビブラート → リバーブ の順に適用

### 音色定義

- GarageBandなどで生成した楽器音のWAVファイルをFFT解析し、倍音構成を参考に波形を再現する
- 対象楽器は以下とする
  - ドラム
  - トランペット
  - フルート
  - クラリネット
- プロトタイプ段階では正弦波単体で代替可能

### 受信〜発音フロー

Processing内での処理順序：

```
1. シリアルポートからバイト列を受信
2. 4バイト固定フレームをパース（type / instrument_id / pitch / velocity）
3. type判定
   - 0x01（NoteOn）  : ボイスを生成してADSRのAttackフェーズへ
   - 0x02（NoteOff） : 対象楽器のボイスをReleaseフェーズへ
   - 0xFF（Failsafe）: 対象楽器のボイスをReleaseへ遷移させ、以降の発音を無効化
4. NoteOn時: instrument_id から楽器の倍音構成テーブルを参照
5. 各倍音について正弦波を生成（周波数 = 基音 × 倍音次数、振幅 = 倍音係数 × velocity）
6. 全倍音の正弦波を加算し、1サンプルの波形値を合成
7. エンベロープ（ADSR）をサンプル単位で適用
   - Attack  : 音の立ち上がり（振幅 0 → max へ線形増加）
   - Decay   : ピーク後の減衰（max → sustain level）
   - Sustain : NoteOff受信まで持続
   - Release : NoteOff後の余韻（sustain level → 0）
8. エフェクト処理
   a. ビブラート（周波数変調：低周波サイン波で音程を揺らす）
   b. リバーブ（遅延加算：ディレイバッファとアッテネーションで残響を付加）
9. 音声出力バッファへ書き込み → サウンドカード出力
```

### NoteOff未着時のタイムアウト

- NoteOffが通信エラーで届かない場合に音が鳴り続けるのを防ぐため、`noteOnTime` を記録する
- NoteOn受信後、**3秒**を超えた場合は自動でReleaseフェーズへ移行する（固定値）

### 音声ライブラリ

- Processingの音声ライブラリは**Minim**を使用する
- Soundライブラリも候補に挙がったが、低レイテンシが期待でき、FFTなどの情報源が多いMinimを採用する

### レイテンシについて

音声出力にはPCのサウンドバッファを介するため、**受信から発音までに遅延が生じる。**
BPMに合わせた音楽再生ではこの遅延が大きすぎると「ずれた演奏」になるため、バッファサイズの調整が重要になる。

| バッファ設定 | 特性 |
|------------|------|
| 大きい | 音切れしにくい・遅延が大きい（数百ms〜秒単位） |
| 小さい | 遅延が小さい・処理が追いつかないと音切れのリスク |

> 許容レイテンシの目標値は**200ms以内**

## フェイルセーフ発動後の挙動

### エラー3回後の動作

- 該当楽器へのCSアサートを停止し、以降そのパートは**無音で演奏継続**
- Processing側へ `type=0xFF`（Failsafe）を送信し、対象楽器のボイスをReleaseへ遷移させる
- 他の楽器ArduinoおよびProcessingへの影響なし（演奏全体はそのまま続く）

### 復帰処理

- **方式A（復帰なし）を採用**：スイッチOFFで全体リセットするまで無音のまま
