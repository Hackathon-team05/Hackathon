プロジェクト概要

目的

Arduino複数台を用いて「かえるの歌」を演奏するオーケストラシステムを構築する。観客の拍手テンポによってBPMをリアルタイム変更し、カエル人形を圧力センサに置くことで手動演奏も可能にする体験型音楽装置。

フェーズ構成

📐 5週：設計（64h）
⚙ 6〜9週：実装・テスト（288h）
🛡 10週：バッファ・デモ（24h+）

1.5 外部インターフェース仕様

① SPI通信（サーバー ↔ 楽器Arduino）

項目仕様
CSピン割り当て楽器1:D4 / 楽器2:D5 / 楽器3:D6 / 楽器4:D7
データ構造SoundData 4バイト固定長
BPM方向サーバー → 楽器：BPMクロック信号（1バイト）
音データ方向楽器 → サーバー：割り込み駆動 SoundData 送信
エラー処理チェックサム方式・連続3回エラーでフェイルセーフ（以降無音）
ハンドシェイク起動時タイムアウト2秒・リトライ3回

② USBシリアル通信（サーバーArduino ↔ PC）

項目仕様
ボーレート115200 bps
データ形式SoundData 4バイト固定長をそのまま転送
Processingハンドシェイク決定：なし（自動受信開始）

③ 圧力センサ入力

項目仕様
ポーリング間隔10ms（暫定）
NoteOn判定アナログ値が閾値を上回った瞬間
NoteOff通知決定：duration=0 のSoundDataを送信
共存方式決定：割り込み方式（該当パートの自動演奏をミュートし手動演奏に切替）

エラー種別検出条件対処復帰方法

4.4 体調不良・作業停止時のリスク対応

⚠ 開発側3名はクリティカルパスに張り付いており、1名の離脱がスケジュールに直結する。以下のルールを必ず徹底すること。

体制ルール

  1. 当日朝8時までに連絡：体調不良で当日作業不能の場合、佐藤（PL）にSlack/LINEで連絡する。
  2. バックアップ担当が即日引継ぎ：表6の「バックアップ」列に記載された担当者が当日中に引き継ぐ。
  3. 2日以上連続離脱の場合：佐藤・藤本が残タスクを再配分し、スケジュールを再調整する（WBS更新）。
  4. 3日以上の場合：10週バッファを前倒し投入して対応。ただしCP5（システムテスト完了）の期限は厳守。

引継ぎを容易にするための義務事項

  • 各実装タスクは「今日どこまで実装したか」を毎日コミットコメントに記載する（例：[WBS:212] SPI初期化まで完了・ハンドシェイク未着手）
  • コード内にTODO / FIXME コメントを残し、未完了部分を明示する
  • 週次ミーティング（毎週月曜）で進捗を口頭共有し、バックアップ担当が作業内容を理解した状態を維持する
  • 重要な設計判断（閾値・ピン番号など）はコード内コメントに理由とともに記載する
